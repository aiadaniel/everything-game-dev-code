# Samples

Complete, playable games generated entirely by the `/full-game` command from this scaffold. No manual coding involved — each game was produced end-to-end from a single natural-language prompt and kept as verbatim harness output.

Conventions for adding and maintaining samples: see [CONVENTIONS.md](CONVENTIONS.md).
Cross-harness analysis of the two controlled experiments: see [harness-comparison.md](harness-comparison.md).

## PrismDefense3D

**Genre:** 3D Tower Defense
**Platform:** Web (HTML5 — Three.js + Vite, ES modules)
**Harness:** Claude Code (June 2026)
**Features:** 12x12 board with fixed S-curve path, 10 scaling waves, 3 tower types with one upgrade level each (sell at 70%), Runner/Tank enemies + wave-10 Boss, orbital camera (drag/pinch/wheel, clamped), mobile-first touch UI with bottom build sheet, strict sim/render split on a fixed timestep (sim is headless — see `scripts/sim-smoke.mjs`), instanced rendering with flat-color primitives, procedural WebAudio SFX + generative music, localStorage high score, GPU memory flat across restarts (first sample exercising the web 3D layer: `rules/web/rendering-3d.md`)

**Deployed:** http://51.75.26.136/prism-defense/ — the URL now serves the
[PrismDefense3DVisuals](#prismdefense3dvisuals) variant (same gameplay, textured visuals);
this original runs locally as generated.

**Prompt (verbatim):**

```
/full-game Game idea: A 3D tower defense called PrismDefense. Enemies march along a fixed winding path across a 12x12 tile board toward the player's crystal base; the player places towers on free tiles to stop them.

Target platform: web (3D — use Three.js).

Scope (keep it exactly this small):

One map with a hardcoded path (S-curve), 10 waves with scaling difficulty.
3 tower types: Cannon (single target, medium damage), Frost (slows enemies in a radius), Laser (fast weak hits, prioritizes strongest enemy). Each has cost, range, fire rate, and one upgrade level (sell refunds 70%).
2 enemy types (Runner: fast/weak, Tank: slow/strong) plus a final Boss on wave 10.
Economy: starting gold, gold per kill, wave-completion bonus. 10 base lives.
Win = survive wave 10. Lose = lives reach 0. High score (waves + gold) in localStorage.
OUT of scope: multiple maps, meta progression, tower targeting menus, pathfinding (the path is fixed).
Camera: orbital rig around the board — drag to orbit, wheel/pinch to zoom, clamped pitch and zoom so the board never leaves the frame. Sensible default angle (~45 degrees) framed for both portrait and landscape.

Mobile-first requirements (must work on a phone):

Touch is a first-class input, not an afterthought: tap a tile to open a radial/bottom build menu, tap to confirm; tap a tower to upgrade/sell. One-finger drag orbits, two-finger pinch zooms. No hover-dependent UI.
Thumb-sized touch targets anchored to safe areas; HUD readable in portrait AND landscape.
Cap devicePixelRatio at 2 and resize the drawing buffer explicitly.
WebAudio unlocks on the first user gesture (same gesture starts the game).
Frame budget defined for a weak phone, not a desktop: instanced or shared geometry for enemies/towers/tiles, shared materials per entity category, no per-frame allocations in the loop, one shadow-casting light maximum (or blob/fake shadows).
Engine layer compliance (this repo's web layer):

Follow rules/web/ including rendering-3d.md: rendering module owns renderer/scene/camera; simulation on a fixed timestep, rendering interpolates; gameplay code never touches Three.js outside the rendering module.
All visuals are placeholder 3D primitives per placeholder-asset-pipeline: boxes/spheres/capsules/cones with one flat-color material per entity category (towers, enemies, path, tiles, base, projectiles) — distinct colors so everything reads instantly. No external assets, no textures.
Procedural WebAudio SFX (build placement thunk, shot, enemy death chirp, wave horn, lose buzz) and a simple generative music loop.
Disposal ownership: restarting a run or returning to the menu must hold GPU memory flat (verify with renderer.info across repeated restarts).
Project conventions:

Generate into samples/PrismDefense3D/ following samples/CONVENTIONS.md: ES modules in src/, runs with a local server, no file:// support, no bundling of src into index.html.
Use Vite with npm run dev for development and npm run build producing a static dist/ with relative base path (base: './') so it serves correctly from the /prism-defense/ subpath.
Record this prompt verbatim in samples/README.md per the conventions.
Deployment (after the game runs locally; we are on Windows/PowerShell):

npm run build, then verify locally with npx serve dist.
Create the target folder and upload dist/ with the verified SSH access:

ssh -i C:\Users\cx02888\.ssh\pirate_deploy root@51.75.26.136 "mkdir -p /var/www/prism-defense"
scp -i C:\Users\cx02888\.ssh\pirate_deploy -r dist/* root@51.75.26.136:/var/www/prism-defense/
Add the nginx location block (matching the existing per-game pattern) to /etc/nginx/sites-enabled/aevora, inside the existing server block, only if it is not already present:

location /prism-defense/ {
    alias /var/www/prism-defense/;
    try_files $uri $uri/ /prism-defense/index.html;
}
Then validate and reload: nginx -t && systemctl reload nginx.
Test from a phone at http://51.75.26.136/prism-defense/.
Acceptance checklist (verify before finishing):

Playable start-to-finish on desktop (mouse) and mobile (touch) — win and lose paths both reachable.
60fps target on desktop, stable frame pacing with 40+ enemies on screen.
renderer.info memory flat across 3 restart cycles.
The deployed URL works from a phone, in portrait and landscape, with audio after the first tap.
```

**Deployment note:** the prompt's nginx instructions assumed host nginx serves the per-game paths; on the VPS those paths are actually served by Caddy (host nginx has been inactive since 2026-04, port 80 is owned by Caddy). The nginx block was added as instructed (validated, harmless), and the equivalent `handle_path /prism-defense/*` block was added to `/etc/caddy/Caddyfile` (backed up first), which is what actually serves the deployed URL.

**Run:** `cd samples/PrismDefense3D && npm install && npm run dev` (ES modules — needs a server; `npm run build` for the static `dist/`, `npm run smoke` for the headless sim test)

## PrismDefense3DAssetsGen

**Genre:** 3D Tower Defense (AI-generated-assets variant of `PrismDefense3DVisuals`)
**Platform:** Web (HTML5 — Three.js + Vite, ES modules)
**Harness:** Claude Code (June 2026), built with the scaffold's `/generate-assets` layer
**Features:** Identical gameplay/sim to `PrismDefense3D`; every visual and audio asset was generated through `manifests/asset-providers.json` (fal.ai, one API key) via `scripts/generate-assets.js` — a 21:9 equirectangular aurora skybox, three seamless tileable board/path/terrain textures, seven Hunyuan3D GLB models (3 towers, 3 enemies, the crystal) produced with a concept-image → Rapid image-to-3D pipeline, ten ElevenLabs sound effects + a CassetteAI battle-music loop, nano-banana-pro menu key art, and a Seedance image-to-video intro cinematic playing behind the menu. Every asset has a `.provenance.json` sidecar (provider, model, prompt, seed, request id) and the raster acceptances live in `generated-assets.json`. GLBs upgrade the instanced primitives asynchronously by swapping geometry/material on the live `InstancedMesh` — instancing, per-instance damage tinting, and the flat-color fallback all survive, so the game stays playable with zero generated files present.

This variant exists as the live demo of the AI asset generation layer: same drop-in-replacement contract as the placeholder pipeline, exercised end-to-end against a real provider.

Second pass (same day): towers now turn toward their current target (view-side aim that mirrors the sim's targeting rules by reading sim state — cannon tracks the furthest enemy in range, laser the highest-hp one, frost idles in a slow spin), and the flat terrain plane became a displaced-relief mesh (board area stays flat) dressed with three more generated environment props (crystal formations, alien spire trees, runed boulders) scattered as `InstancedMesh` rings around the board.

Third pass: two selectable levels. `LEVELS` in `sim/config.js` owns per-level paths (`buildPath` parameterized, `sim.reset(levelIndex)`), and the render layer mirrors the index with per-level themes (`setLevel`: board rebuild from waypoints, texture/skybox/fog/light swap, prop-set visibility). Level 2 "Mars Outpost" has a longer zigzag path and a fully generated Martian set — butterscotch-haze skybox, red rock/regolith/dust textures, hoodoo spire and rock arch props. The menu gained level-select buttons and pause/end screens gained a Main menu button.

**Deployed:** http://51.75.26.136/prism-defense-gen/ (the original variants remain at `/prism-defense/`)

**Run:** `cd samples/PrismDefense3DAssetsGen && npm install && npm run dev` (`npm run smoke` still passes — the sim is untouched)

## NebulaLance

**Genre:** 2.5D side-scrolling shoot-'em-up (R-Type homage)
**Platform:** Web (HTML5 — Three.js + Vite, ES modules)
**Harness:** Claude Code (June 2026), built by one `/full-game` run with the `/generate-assets` layer
**Features:** A horizontal auto-scrolling shmup with the two R-Type signatures — the **Wave Cannon** (hold FIRE to charge a piercing beam whose damage scales with charge) and the **Force** pod (a detachable, invulnerable orbital drone that attaches front/rear, blocks enemy fire, launches/recalls, and cycles spread/laser/homing weapons). One authored stage rises from trash-mob waves through a closing-corridor turret gauntlet to a three-phase boss, **THE MAW**, whose only weak point is a destructible glowing core per phase. Four enemy archetypes (gunner/weaver/carrier/turret) + boss minions, four power-ups (force/power/speed/shield), combo multiplier, 3 lives, localStorage high score. Mobile-first: finger-follow movement + on-screen FIRE/FORCE/LAUNCH buttons, plus full keyboard. Strict sim/render split on a fixed timestep with a headless sim smoke test (`npm run smoke` asserts a lose path and an auto-aim bot clearing all three boss phases), instanced rendering, and GPU memory flat across restarts (verified: 18 geometries / textures / programs constant).

Every visual and audio asset is AI-generated through `manifests/asset-providers.json` (fal.ai) via `scripts/generate-assets.js`: seven Hunyuan3D models (ship, Force, boss, four enemies) from a concept-image → Rapid image-to-3D pipeline, a nano-banana-pro menu key-art still, a Seedance **image-to-video** intro cinematic playing behind the menu, a parallax nebula backdrop, ten ElevenLabs sound effects, and a CassetteAI battle-music loop. Each asset carries a `.provenance.json` sidecar; raster acceptances live in `generated-assets.json`. Models swap onto the live primitives in place, so the game is fully playable on flat-color placeholders with zero generated files present — generation is the optional upgrade gated by the cost-confirmation policy (`rules/common/asset-pipeline.md`).

**Run:** `cd samples/NebulaLance && npm install && npm run dev` (`npm run smoke` for the headless sim test, `npm run build` for the static `dist/`)
